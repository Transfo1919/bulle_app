# Bulle — Installation & Configuration

## Prérequis

- **Node.js** 18+ et npm
- Un projet **Supabase** (offre gratuite suffisante)

## Installation locale

```bash
git clone https://github.com/Transfo1919/bulle_app.git
cd bulle_app
npm install
```

## Configuration Supabase

1. Créer un projet sur [supabase.com](https://supabase.com)
2. Récupérer l'URL du projet et la clé **anon public** (Settings → API)
3. Exécuter le script **`supabase_schema.sql`** (à la racine du repo) dans Supabase → SQL Editor.
   Il complète les tables existantes (`moments`, `bucket_items`, `prayer_topics`, `game_sessions`)
   et crée les tables manquantes (`collections`, `custom_questions`, `custom_defis`).
4. Créer un bucket de stockage nommé **`photo_moment`**, coché **Public**, dans Supabase → Storage.

### Variables d'environnement

Créer un fichier `.env.local` à la racine (voir `.env.example`) :

```
VITE_SUPABASE_URL=https://votre-projet.supabase.co
VITE_SUPABASE_ANON_KEY=votre_clé_anon
```

Sur Vercel, ajouter ces deux mêmes variables dans Project Settings → Environment Variables
(pour Production, Preview et Development), puis redéployer.

## Lancer en local

```bash
npm run dev
```

Accessible sur `http://localhost:5173`.

## Build & déploiement

```bash
npm run build     # build de production dans dist/
npm run preview   # prévisualiser le build localement
```

Le déploiement se fait automatiquement via Vercel à chaque push sur `main`.

## Schéma des données (résumé)

| Table | Rôle |
|---|---|
| `moments` | Les Instants (souvenirs) |
| `bucket_items` | Les Envies (`timing`: ce_soir / plus_tard) |
| `prayer_topics` | Prière (`category`: toport / recognition) |
| `game_sessions` | Historique des parties "À deux" |
| `collections` | Regroupements d'instants |
| `custom_questions` | Questions ajoutées par les utilisateurs (jeu "Qui connaît le mieux ?") |
| `custom_defis` | Défis ajoutés par les utilisateurs |

Toutes les colonnes `id` sont en génération automatique (`bigint generated by default as identity`) —
l'application n'envoie jamais d'id manuellement à la création.

## En cas de conflit Git

Si deux personnes poussent sur `main` en parallèle, préférez des branches séparées et une
Pull Request GitHub pour repérer les conflits visuellement, plutôt qu'un merge en ligne de
commande qui peut laisser des marqueurs `<<<<<<<` non résolus silencieusement.
